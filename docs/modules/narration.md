# Narration Module

## Purpose

Handles narration audio metadata and timing synchronization. Three parts:

1. **Manifest** — data format for per-clip narration metadata (`narration-manifest.ts`)
2. **Sync engine** — legacy estimate-based step-padding engine (`narration-sync.ts`, optional)
3. **Auto-shift** — post-recording placement correction in `processVideoWithAudio()` (default)

## Location

- `src/narration-manifest.ts` — Manifest data format
- `src/narration-sync.ts` — Legacy estimate-based timing sync engine (exported, not in default pipeline)
- `src/stages/sync.ts` — Optional NarrationSyncStage for custom pipeline compositions
- `src/video-handler.ts` — Auto-shift logic in `processVideoWithAudio()` (default behavior)

## Narration Manifest (`narration-manifest.ts`)

### Purpose

Defines the on-disk format for per-clip narration metadata. The manifest is generated by the TTS stage and consumed by the audio mix stage.

### Schema

```ts
export const narrationManifestClipSchema = z.object({
  sceneIndex: z.number().int().min(0),
  stepIndex: z.number().int().min(0).optional(),
  narration: z.string().min(1),
  filePath: z.string().min(1),
  audioDurationMs: z.number().int().min(0),
  audioOffsetMs: z.number().int().min(0).optional(),
  gapAfterMs: z.number().int().min(0).optional(),
});

export const narrationManifestSchema = z.object({
  version: z.literal(1),
  processingVersion: z.string().min(1),
  audioPath: z.string().min(1).optional(),
  clips: z.array(narrationManifestClipSchema).min(1),
});

export const NARRATION_PROCESSING_VERSION = "v5-no-volume-normalization";
```

### Clip Fields

| Field             | Type    | Description                                          |
| ----------------- | ------- | ---------------------------------------------------- |
| `sceneIndex`      | number  | Index into `config.scenes` array                     |
| `stepIndex`       | number? | Step index this narration belongs to                 |
| `narration`       | string  | The original narration text                          |
| `filePath`        | string  | Relative path to individual scene MP3 clip           |
| `audioDurationMs` | number  | Duration of the generated audio in milliseconds      |
| `audioOffsetMs`   | number? | Optional offset within the full narration track      |
| `gapAfterMs`      | number? | Minimum gap after narration before next scene begins |

### Processing Version

`NARRATION_PROCESSING_VERSION = "v5-no-volume-normalization"`

When the TTS processing changes (different model, different settings, different audio format), this version bumps, triggering regeneration of all cached narration artifacts.

### Path Helpers

```ts
export function getNarrationManifestPath(audioPath: string): string;
// "output/demo-narration.mp3" → "output/demo-narration.manifest.json"

export function getNarrationClipDir(audioPath: string): string;
// "output/demo-narration.mp3" → "output/demo-narration-clips/"

export function getNarrationClipFileName(sceneIndex: number): string;

```

### File Layout

```
output/
  onboarding-narration.mp3           ← Full concatenated narration track
  onboarding-narration.manifest.json ← Per-clip metadata
  onboarding-narration-clips/
    scene-000.mp3                    ← Individual scene clips
    scene-001.mp3
    scene-002.mp3
```

## Auto-Shift (`video-handler.ts`)

### Purpose

The default pipeline uses **post-recording auto-shift** instead of the estimate-based sync engine. After the demo is recorded, `processVideoWithAudio()` builds `NarrationPlacement[]` from real scene timestamps. If adjacent placements overlap, the later one is shifted forward to start at the previous one's end. This cascades so all subsequent placements stay non-overlapping.

### Three Modes

| Mode     | Behavior                                                |
| -------- | ------------------------------------------------------- |
| `auto`   | Shift overlapping placements forward to prevent overlap |
| `strict` | Throw an error if any overlap is detected               |
| `off`    | Passthrough — no overlap handling                       |

### Algorithm

```
For each adjacent pair of narration placements (sorted by startMs):
  If current.startMs < previous.endMs:
    overlap = previous.endMs - current.startMs
    If mode is "off": skip
    If mode is "strict": throw Error
    Else (auto): current.startMs = previous.endMs; current.endMs += overlap
    Log: "Narration auto-shift: scene N pushed forward Xms"
```

Subtitles and metadata use the corrected `narrationPlacements`, so the shift
propagates through to output files.

### Error Handling

- **`auto` mode:** Fixes overlaps silently, logs a warning for informational purposes.
- **`strict` mode:** Throws `Error` with details about the overlapping scenes.
- **`off` mode:** No handling, placements may overlap.

## Narration Sync Engine (`narration-sync.ts`) — Legacy

### Purpose

**Not used in the default pipeline.** This is the legacy estimate-based step-padding engine.
Use `NarrationSyncStage` from `src/stages/sync.ts` for custom pipeline compositions that
need pre-recording step timing adjustment.

### Three Modes

| Mode     | Behavior                                                         | When to use      |
| -------- | ---------------------------------------------------------------- | ---------------- |
| `auto`   | Pads step delays to cover narration duration. Warns on overflow. | Normal operation |
| `strict` | Fails if any scene window can't cover narration duration.        | CI/CD validation |
| `off`    | No sync — passthrough. Steps run at their natural speed.         | Dry runs, tests  |

### Scene Window Model

Each scene owns all steps from its `stepIndex` up to (exclusive) the next scene's `stepIndex`:

```
Example: scenes at [4, 6] with 10 total steps
  Scene 0 window: steps 4..5  (2 steps)
  Scene 1 window: steps 6..9  (4 steps)

Each window's estimated duration must be >= narration duration + gap
```

### Step Duration Estimation

Default estimates (mirrors `script/timing.ts`):

| Action                      | Estimated Duration             |
| --------------------------- | ------------------------------ |
| `goto`                      | 2000ms                         |
| `click`, `hover`            | 700ms                          |
| `type`                      | `text.length × 100ms` per char |
| `press`                     | 200ms                          |
| `scroll`                    | 400ms                          |
| `select`, `check`, `upload` | 300ms                          |
| `drag`                      | 1000ms                         |
| `wait`                      | `step.ms`                      |
| `waitFor`                   | 500ms                          |
| default                     | 500ms                          |

Plus `delayBeforeMs` + `delayAfterMs` if set.

### SyncConfig

```ts
export interface SyncConfig {
  narrationSyncMode: "auto" | "strict" | "off";
  narrationGapMs: number; // Default gap after narration (300ms normal, 200ms fast)
  maxAutoPadMs: number; // Maximum auto-padding before warning (5000ms normal)
  maxSyncPasses: number; // Maximum sync iterations (2 normal, 1 fast)
}
```

### SyncReport

```ts
export interface SyncReport {
  windows: SceneWindow[]; // Per-scene window analysis
  totalDeficitMs: number; // Total padding needed
  maxDeficitMs: number; // Largest single-scene deficit
  overflowScenes: number[]; // Scenes exceeding maxAutoPadMs
  appliedPadMs: number; // Total padding applied
}

export function logSyncReport(report: SyncReport, verbose?: boolean): void;
```

### Padding Strategy

When a scene window has a deficit (steps are too fast to cover narration):

1. **If last step is `wait`** → Increase `wait.ms` by deficit
2. **If last step supports `delayAfterMs`** → Add deficit to `delayAfterMs`
3. **Otherwise** → Insert a new `{ action: "wait", ms: deficit }` step at window end

After padding, downstream scene `stepIndex` values are shifted to account for any inserted steps.

### Error Handling

- **`auto` mode:** Pushes overflow scenes to `SyncReport.overflowScenes`. Caller can decide to warn or continue.
- **`strict` mode:** Throws `Error` with detailed per-scene deficit breakdown.
- **`off` mode:** Skips entirely, returns unmodified steps.
